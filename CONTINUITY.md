## 2026-05-07 — Cron-режим через Windows Task Scheduler

### Что добавлено

**Цель:** автоматизировать ежедневный запуск дайджестов (утром 08:03 и вечером 19:03) с максимальной приватностью — без облака, всё локально на ПК пользователя.

### Новые файлы

- **`make_digest.py`** — одноразовая точка входа: инициализирует БД + Telethon + минимальный aiogram Bot (без polling), вызывает `check_and_send_digests(force=True)`, выходит. Pre-flight проверка Telethon-сессии до `init_client()` — защита от зависания на `input()` в Task Scheduler.
- **`run_digest.bat`** — обёртка для Task Scheduler: `chcp 65001` + `cd /d "%~dp0"` + `venv\Scripts\python.exe make_digest.py`. Корректно работает с кириллицей в пути проекта.
- **`register_tasks.ps1`** — регистрирует 3 задачи в Windows Task Scheduler (BOM UTF-8 обязателен — иначе PowerShell 5.1 ломает парсер на кириллических комментариях).

### Изменения в существующем коде

- **`src/scheduler/tasks.py`** — добавлен опциональный параметр `force: bool = False` в `check_and_send_digests` и `_process_user_digest`. Default-значение сохраняет старое поведение → интерактивный режим (`src/main.py`) и `catch_up_digests()` не затронуты.

### Расписание

| Задача | Триггер | Назначение |
|---|---|---|
| `TG_Digest_Morning` | ежедневно 08:03 | утренний дайджест |
| `TG_Digest_Evening` | ежедневно 19:03 | вечерний дайджест |
| `TG_Digest_OnLogon` | при входе в Windows + 1 мин | подхват пропущенных запусков |

### Покрытие сценариев

- **ПК работает в 8:03/19:03** → штатное срабатывание
- **ПК выключен в 8:03, включаем в 9:30** → `StartWhenAvailable` запускает пропущенную задачу через ~1 мин после загрузки; параллельно срабатывает `OnLogon` — антидубль (2ч) пропускает повтор
- **ПК в режиме сна** → `WakeToRun` будит и запускает
- **ПК выключен весь день** → `OnLogon` отрабатывает при следующем включении

### Антидубль (внутри `make_digest.py`)

Маркер `data/last_digest_run.txt`. Если прошло < 120 мин с последнего успешного запуска — скрипт выходит без работы. Флаг `--force` игнорирует антидубль (используется при ручном тестировании).

### Решения

- **Почему Task Scheduler, а не GitHub Actions?** Требование пользователя «теоретически нет доступа к данным». GHA = секреты в GitHub Inc., session-файл в репо. Task Scheduler = всё на локальном ПК.
- **Почему `force=True` в cron-режиме?** Расписанием управляет Task Scheduler снаружи. `user.digest_interval` в БД оставлен для интерактивного режима (там пользователь сам настраивает периодичность через бота).
- **Почему `LogonType Interactive` (а не `S4U`)?** S4U требует сохранения пароля в LSA. Interactive — без пароля, но задачи фактически работают только когда пользователь залогинен. Для домашнего ПК это норма.
